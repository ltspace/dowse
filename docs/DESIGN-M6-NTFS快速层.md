# 里程碑 6 设计说明：NTFS 快速层（MFT 直读 + USN Journal）

目标一句话：在 NTFS 卷上，首次建文件名索引从"分钟级爬目录"变成"秒级读账本"，
错过的变更从"全盘 mtime 对账"变成"读日志流水补账"。Everything 的两板斧,
接进我们已有的三层索引架构。

## 一、与现有架构的关系

里程碑 3 已经留好了插座：事件源只需产出 `WatchEvent`（Upsert / Remove /
RemoveDir / Rename）四种归一事件，防抖、合并、水位、增量更新全部复用。
本里程碑做三件事：

1. **MFT 快速枚举**。初次扫描不走 walkdir 爬目录，直接读 NTFS 的主文件表；
2. **USN 事件源**。运行期监听从 notify 换成读 USN Journal（notify 保留为降级路径）；
3. **USN 游标补账**。启动对账优先按游标回放日志，mtime 全扫降为兜底。

```
                     ┌── 有管理员权限 + NTFS ──▶ MFT 枚举 + USN 事件源 + 游标补账
  每个监听根按卷判定──┤
                     └── 否则 ──────────────────▶ walkdir + notify + mtime 对账（现状）
```

降级判定按卷、在运行时做，两条路径产出的事件和文档完全一致。上层感知不到差别。
诚实降级：落到哪条路径、为什么，写日志（材质降级那次的教训）。

## 二、MFT 快速枚举

- 用卷句柄发 USN 枚举控制码（FSCTL_ENUM_USN_DATA）遍历全卷文件记录。这是
  Everything 同款技术：不打开任何文件、不走目录树，顺序读出每条记录的
  文件参照号（FRN）、父 FRN、文件名。
- 内存里按 FRN → (父 FRN, 名字) 建表，从叶子回溯拼出完整路径；只保留落在
  已注册监听根内的条目（全盘模式本里程碑不做）。
- 拼出来的清单喂给现有索引管线。MFT 记录不含可靠的 mtime/size，
  走"文件名先入索引 + 内容抽取后台补"的既有节奏，这正是三层索引
  "逐层变重"设计预留的形态。
- 需要管理员权限拿卷句柄。没有就整条路径不可用，静默走 walkdir，日志记一行。

## 三、USN 事件源

- 按卷起一个读取线程，FSCTL_READ_USN_JOURNAL 阻塞式拉取记录，翻译成
  WatchEvent 喂进现有防抖队列：
  - 创建/写入/属性变更 → Upsert
  - 删除 → Remove（目录记录带目录标志 → RemoveDir）
  - 重命名是**两条记录**（旧名 + 新名，同一 FRN）：配对成 Rename{from,to}；
    配不上对的单边按 in_scope 规则自然退化（防抖队列已处理）
- **已知的坑，设计期就写进测试清单**：同一 FRN 上"重命名后紧跟删除"这类
  快速连续操作，记录配对状态机容易吞事件——单测要覆盖乱序/交错序列。
- USN 记录给的是 FRN + 文件名，不是全路径：维护 FRN → 路径 的缓存表
  （MFT 枚举时顺手建好，收到目录事件时增量维护）。缓存 miss 时用
  FRN 打开句柄反查路径兜底。

## 四、游标补账

- meta.json 每卷存两个数：Journal ID + 下一个 USN 位置，每批 commit 成功后更新。
- 启动时：Journal ID 对得上且游标仍在日志保留范围内 → 从游标回放到当前,
  秒级追平;Journal 被重建/回绕（ID 变了或游标过期）→ 退回 mtime 全扫对账。
- 游标持久化与批量 commit 的原子性：先 commit 索引、后写游标。宁可重放一批
  （Upsert 幂等），不可漏一批。

## 五、性能预算

| 指标 | 预算 |
|------|------|
| MFT 枚举 100 万条记录 | < 5s |
| 文件名层可搜（10 万文件的根） | < 10s（内容后台继续补） |
| USN 游标回放 10 万条变更 | < 10s |
| USN 读取线程常驻内存（含 FRN 路径表） | < 80MB / 百万条目 |

## 六、验收清单

1. 管理员运行，对一个 10 万文件的 NTFS 根建索引：文件名层 10 秒内可搜。
2. 非管理员运行同样操作：自动走 walkdir 路径，功能等价，日志明确说明降级原因。
3. 停掉 dowse，改/删/增/重命名各若干文件，重启：USN 回放追平，
   不触发全盘 mtime 扫描（日志验证）。
4. 快速连续"改名→删除"同一文件：索引终态正确（旧名新名都搜不到）。
5. FAT32/exFAT 卷（U 盘）上的监听根：自动落 notify 路径，不报错。
6. 游标过期场景（人为改 meta.json 的 USN 为过期值）：正确退回 mtime 对账。

## 七、明确不做（本里程碑）

- 全盘文件名模式（不注册根、直接搜整个 C 盘）：留给以后的"Everything 模式"
  产品决策，本里程碑 MFT/USN 只服务已注册的监听根
- 非 NTFS 文件系统的任何快速路径
- ReFS 支持（它有自己的变更日志,几乎没人在桌面用,不值当）

## 八、实施备注

以下是实现过程中的落地细节和对设计的补充，跟本文档前七节的关系是"如何做到"
而不是"要不要改主意"。没有偏离上面任何一条设计决策。

**FRN 表怎么在"MFT 枚举"和"USN 事件源"之间接力。** 每次快车道启动
（首次建索引，或托盘程序/CLI 重新挂监听）都会重新做一次 MFT 枚举来建 FRN
表。内存里的旧表不跨进程重启存活，也没有必要保存：枚举本身很快（性能预算
100 万条 < 5s），比"设计一套序列化格式把表存盘再校验它没过期"更简单可靠。
游标补账的价值不在于省下 MFT 枚举，而在于省下"每个文件都要跟索引比一遍
mtime/size"这一步。游标有效时，MFT 枚举只重建路径表，真正决定"要不要
碰这个文件"的是 USN 日志里精确的变更记录，不是全量比对。

**MFT 枚举先建整卷临时表、再瘦身。** FSCTL_ENUM_USN_DATA 按 FRN 顺序返回
记录，父子记录谁先谁后没有保证，必须先把整卷记录收全才能可靠地做父链
回溯；`FrnTable::retain_in_scope()` 枚举完毕后一次性瘦身，只留下监听根子树
内的条目。常驻内存的表大小只跟"监听根下有多少文件"成正比，不跟"整卷有
多少文件"成正比，这是性能预算里内存上限（80MB/百万条目）的前提。

**游标持久化的原子性怎么落地（"先 commit 索引、后写游标"）。** 实时监听阶段，
USN 读取线程和 `run_watch` 的主循环分别在不同线程：读取线程把翻译好的事件
送进防抖队列，主循环单线程消费、攒够一批才 commit。`CursorSync`
（cursor.rs）用一把锁把"登记这个事件对应的 usn"和"把事件真正送进 channel"
绑成同一个临界区。配合 `run_watch` 本来就有的 `Received`/`Committed`
进度回调，这把锁保证游标只能在对应的那批变更真正提交成功之后才被标记为
"可以持久化"。多卷并发读取线程共用同一个 channel 时，这把锁也保证了
"登记顺序"和"事件送达顺序"一致，不会张冠李戴到别的卷头上。游标补账
（启动时的一次性回放，不是常驻监听）不需要这套机制，因为它是同步执行、
一次 commit 到底，天然满足"先 commit 再更新游标变量"的顺序。

**"重命名后紧跟删除"这个坑具体怎么防的。** 状态机（usn_translate.rs）的
不变式是"FrnTable 在每条记录处理时立刻更新到这条记录反映的最新状态，绝不
滞后"。NEW_NAME 记录到达时立刻把表指向新位置，而不是等配对确认后再更新。
这样即使删除记录紧跟着到达，用表查出来的路径已经是新名字，不会删错地方
留下孤儿文档。反过来，OLD_NAME 记录到达、还没等到 NEW_NAME 就先等到删除的
（更罕见的乱序情形）会用配对时记下的旧路径兜底发一次删除，不会吞事件。
单测覆盖了这两种顺序，以及两个不同 FRN 的重命名交错到达的情形
（usn_translate.rs::tests，尤其是
`rename_immediately_followed_by_delete_before_pairing_completes_is_not_swallowed`
和 `rename_completes_then_delete_targets_the_new_path_not_the_old_one`）。

**权限探测降级的信号，就是"打开卷句柄成功与否"本身，不额外查进程 token。**
`\\.\C:` 这样的原始卷句柄需要管理员（或备份操作员）权限才能打开，这个
事实本身就是最诚实的降级依据。不需要另外调 `GetTokenInformation` 查
`TOKEN_ELEVATION`，省一层容易跟真实行为脱节的间接判断。

**每卷的探测结果缓存到进程生命周期，不是每次都重新试。** 本机测试环境中，
非管理员权限下反复调用 `CreateFileW(\\.\C:, ...)`（每次都正确失败并降级）
偶尔会在同一进程内造成对完全无关的文件（tantivy 索引的临时分段文件）写入
被瞬时拒绝访问。复现率大约三分之一，不是持续加重的资源泄漏模式；已确认
这台机器的 Windows Defender 实时防护处于关闭状态，排除了最常见的"杀软扫描
锁文件"解释，真正根因没有继续深挖——不影响功能正确性，只是探测这个动作本身
在某些安全软件/驱动栈上可能有超出预期的副作用。鉴于"同一个卷的管理员权限
状态在进程生命周期内不会变"这条本来就成立，把每卷的探测结果缓存起来是
无条件安全的优化，顺带把这个不完全确定成因的副作用面降到最小，不需要先
查清楚根因才能规避它。详见 `volume.rs::CAPABILITY_CACHE` 的注释。

**已知限制、留给以后的机会。**
- MFT 枚举/游标补账目前按"每个监听根各自处理"的粒度调用；同一个卷上有
  多个监听根时，会各自打开卷句柄探测（探测结果因为上面的缓存只算一次），
  但 MFT 枚举/游标补账仍是每个卷一次（`bootstrap_fast_roots` 已经按卷分组），
  这点跟设计一致，不是限制，写在这里是为了跟"每个根各自处理"的表述做区分。
- 游标补账重放期间的变更走的是内存里的 `Debouncer` 合并后一次性 `apply`，
  不是像实时监听那样每 500ms 刷一批——这是有意为之（补账不需要防抖，越快
  追平越好），跟"游标持久化只能在 commit 成功后才推进"的原子性要求不冲突。
- 本机开发环境不是管理员权限，快车道（MFT 枚举 + USN 事件源）的真实端到端
  路径没有在真机上验证过（详见任务报告"权限探测结果与真机验证程度"一节）；
  降级路径（本文档验收清单第 2 条）已经在真实文件系统操作下反复验证。
