# 里程碑 4 设计说明：OCR 管线

目标一句话：截图和图片里的文字进索引，比如搜"限流"就能命中上个月存的那张架构图。

技术验证已完成（2026-07-11 spike），本文档的所有数字来自实测，不再是估算。

## 一、验证结论回放

| 问题 | 结论 |
|------|------|
| Rust 调 Windows 自带 OCR | 通，windows crate，零安装依赖 |
| 中文语言包 | 系统自带 zh-Hans-CN，且同一引擎覆盖英文/数字，无需额外装包 |
| 识别质量 | 正常字号中文准确率 95%+；中英混排（代码、标识符）接近零错 |
| 速度 | 1080p 中密度截图约 112ms/张（单线程中位数）；稀疏小图 10ms 量级 |
| 系统性误差 | 左右结构汉字偶发拆分（池→氵也、语→讠吾言），无法自动纠正，接受 |

## 二、管线设计

```
  扫描/监听发现图片 ──▶ OCR 队列（持久化） ──▶ worker 池（2~4 线程）
                                                │  每 worker 独占一个 OcrEngine
                                                ▼
                                          后处理清洗
                                                │  仅删 CJK 字符间空格，
                                                │  英文单词间空格保留
                                                ▼
                                          入索引（双形态）
```

- **独立于文本管线**：文本抽取快（毫秒级）走主流水线；OCR 慢（百毫秒级）走
  独立的低优先级队列。全量建索引时先让文本部分完成可搜，图片在后台慢慢消化，
  与"三层索引逐层变重"的总设计一致。
- **worker 池 2~4 线程**：OcrEngine 与 SoftwareBitmap 均非线程安全，
  每个 worker 独占引擎实例，绝不跨线程共享。按 112ms/张估算，
  4 worker 吞吐约 30 张/秒，一万张截图约 6 分钟消化完。
- **队列持久化**：进度落盘（已处理文件列表并入对账机制），程序中途退出
  不从头再来。与里程碑 3 的启动对账共用 (path, mtime, size) 三元组判断增量。
- **异步调用**：生产代码不用 spike 里的 busy-poll，用 pollster 或接入
  现有 async 运行时消费 IAsyncOperation。

## 三、索引侧变化

- **schema 升级**：新增 kind 字段（text / image，STORED）。版本号在
  里程碑 3 的 v2 基础上升到 v3，沿用 meta.json 版本检查机制，不静默兼容。
- **双形态入索引**（对冲拆字误差与分词间隙）：OCR 原始输出和
  "去 CJK 间空格拼接"两种形态都写入 content，用空间换召回。
- **范围**：png / jpg / jpeg / webp / bmp；单文件上限沿用 20MB；
  OCR 结果为空的图片记为"已处理无文本"，不重复重试。
- **前端呈现**：结果行给图片缩略图；预览区显示原图 + 命中的 OCR 文本段；
  Ctrl+Enter 跳转文件夹行为与文本文件一致。

## 四、降级与错误处理

- 系统无任何 OCR 语言包（理论上 Win11 中文系统不会发生）：管线整体停用，
  托盘与浮窗引导页提示一句话，不报错不崩溃。
- 单张识别抛错：跳过并计数，不中断队列。
- 语言引擎创建用 TryCreateFromUserProfileLanguages，不做语言标签精确匹配
  （spike 教训：标签是 zh-Hans-CN，写死 zh-Hans 会匹配失败）。

## 五、性能预算

| 指标 | 预算 |
|------|------|
| 单张 1080p 截图 OCR | < 200ms（实测 112ms，留余量） |
| 一万张截图后台消化 | < 10 分钟（4 worker） |
| OCR 期间对搜索延迟的影响 | 不可感知（低优先级线程 + 批量 commit） |
| worker 池常驻内存增量 | < 60MB |

## 六、验收清单

1. 往被监听目录丢一张含"独角鲸测试标记"文字的截图，一分钟内可搜到。
2. 中文、英文、中英混排三种截图各若干张，命中率抽查 ≥ 90%。
3. 建索引期间连续搜索，延迟无可感知劣化。
4. 程序在 OCR 队列半程时退出再启动，队列从断点继续，不重复处理。
5. GBK 命名 + 中文路径的图片正常处理（路径编码不踩坑）。

## 七、明确不做（本里程碑）

- PDF 内嵌扫描页的 OCR（属文档抽取管线的加强，另立条目）
- 手写体、倾斜矫正、放大预处理（spike 证明正常截图不需要；
  小字号截图质量不满意时再评估"放大 2 倍"选项）
- 云端 OCR 后备（违背"全程本地"原则，永不做）

## 八、实施备注（对照本文档的偏离与补充）

- **schema 升级本轮没做**：第三节写的"新增 kind 字段、版本号升到 v3"这部分
  推迟到身份模型分支合并阶段统一处理，避免两个并行分支抢同一份 schema
  定义。图片文档当轮复用 v2 的字段（path/name/ext/content/mtime/size），
  没有 kind 区分文本和图片。
  （后续更新：kind 字段已随 v3 schema 落地——文本文档写 "text"、图片文档写
  "image"，见 `lib.rs` 的 schema 定义；当时留在
  `indexer.rs::add_image_document_with_content` 的接线 TODO 注释已随实现移除，
  本条仅存档当轮的偏离记录。）
- **结果行缩略图本轮没做**：第三节"前端呈现"提到的"结果行给图片缩略图"
  延后。ResultRow 组件当时正被另一个分支改动，避免冲突。预览区（选中后
  的原图 + OCR 文本）已经实现。
- **OCR 队列的"已处理"缓存连内容一起存，不只是个标记**：全量重建索引会把
  tantivy 索引整个删掉重建（`indexer.rs::rebuild_index`），如果"已处理"只是
  个布尔值，每次全量重建都要把所有图片重新 OCR 一遍。这超出了设计文档"队列
  持久化"一句话的字面意思，但不这么做，全量重建的代价完全不可接受。
  实际实现：`<index_dir>-ocr-queue.json` 里 `processed` 表存的是
  `(mtime, size) -> 双形态清洗后的 content`，重建时命中缓存直接复用，不重新
  识别；只有真正新增/变化的图片才进 `pending` 排队。
- **OCR 结果写回的 commit 粒度是"每张图片一次"，不是时间窗口批量提交**：
  第二节"批量 commit"的字面表述是按窗口攒批；实际实现里每个 worker
  识别完一张、写回索引后立即单独 commit（见
  `updater.rs::stage_and_commit_ocr_result`）。理由：写回操作在 tantivy
  层很轻，OCR 识别本身（百毫秒级）才是真正的开销，单文档 commit 不会成为
  瓶颈；换来的是实现简单、每识别完一张就立刻可搜，不会因为 worker 中途
  异常退出丢掉一整批还没提交的结果。如果实测发现 commit 频率确实影响到了
  性能预算，是后续容易调整的一处（合并成按数量或按时间窗口的批量提交）。
- **`\\?\` 扩展长度路径踩坑**：真机验证阶段发现，`Path::canonicalize()`
  在 Windows 上产出的 `\\?\C:\...` 扩展前缀路径，喂给
  `StorageFile::GetFileFromPathAsync` 会报"指定的路径过长"。不是真的路径
  太长，是 WinRT 的路径解析根本不接受这种 Win32 扩展前缀语法。`dowse index`
  这条命令行入口恰好会走 `canonicalize()`，三张验证图片全部识别失败，
  已在 `ocr.rs::strip_extended_prefix` 里剥前缀修复，并补了单测。
