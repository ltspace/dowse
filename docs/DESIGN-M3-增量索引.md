# 里程碑 3 设计说明：文件监听与增量索引

目标一句话：改一个文件，几秒内就能搜到新内容；从此告别全量重扫。

## 一、现状与要补的洞

里程碑 1 的索引是一次性快照：`dowse index <目录>` 全量重建，文件后续的增删改
索引一概不知。要补三件事：

1. **感知变化**：文件系统监听
2. **增量更新**：只动变了的文档，不推倒重来
3. **断线重连**：程序没开着的时候错过的变化，启动时补账

## 二、总体结构

```
                         ┌──────────────────────────────────┐
                         │  dowse-app（常驻进程）              │
                         │                                  │
   文件系统事件 ──────────▶│  watcher ──▶ 事件队列 ──▶ 更新器    │──▶ tantivy 索引
   (notify 库)            │            (防抖合并)   (增删文档)  │
                         │                                  │
   启动时 ────────────────▶│  对账器（mtime 快扫，补错过的账）    │
                         └──────────────────────────────────┘
```

- 监听和更新逻辑全部放 **dowse-core** 新模块里，dowse-app（托盘常驻）是宿主；
  CLI 加 `dowse watch` 命令用于无 UI 调试。
- 事件源本里程碑用 **notify 库**（跨平台、够用）；NTFS USN Journal 是里程碑 6
  的快速路径，接口设计上给它留位置（事件来源抽象成 trait）。

## 三、关键设计决策

### 1. path 是文档主键
索引里 path 字段本来就是 STRING（不分词原样存）。增量更新的原语：

| 事件 | 索引操作 |
|------|---------|
| 新建/修改 | `delete_term(path)` + 重新抽取 + `add_document`（先删后加，天然幂等） |
| 删除 | `delete_term(path)` |
| 重命名 | 删旧 path + 按新 path 加（notify 给 from/to 就精确处理，只给一边就当删/增） |
| 目录删除/重命名 | 前缀圈选：对 path 字段做 `[前缀, 前缀+U+10FFFF)` 的 term 范围查询，收集命中文档逐个删 |

### 2. 防抖与批量提交
编辑器保存一次可能触发 3~5 个事件（写临时文件、改名、改属性），编译/下载
一秒能产出几百个事件。策略：

- 事件进队列，**500ms 静默窗口**合并同一 path 的事件（保留最后状态）
- 窗口关闭后批量处理，一批一次 `commit`（tantivy 的 commit 是重操作，
  绝不能一个文件一次）
- 队列水位超阈值（比如 5000）时立即处理一批，防内存膨胀

### 3. 启动对账：补错过的账
程序没跑的时间里文件照样在变。启动时对每个已注册的索引根：

- 遍历文件系统拿 (path, mtime, size)，和索引里存的比对
  （schema 要新增 mtime、size 两个 STORED 字段——顺手给搜索结果展示用）
- 三类差异：磁盘有索引无 → 增；两边都有但 mtime/size 变了 → 改；
  索引有磁盘无 → 删
- 对账在后台低优先级跑，期间索引照常可搜（旧数据可搜好过不可搜）

### 4. 元数据落盘
索引目录旁放一个 meta.json：schema 版本号 + 已注册的索引根列表。
- schema 版本不匹配（升级了字段定义）→ 提示需要重建，不静默兼容
- 索引根列表是对账器和托盘菜单"重建索引"的依据

### 5. 错误处理原则
- 单个文件抽取失败：跳过并计数，不中断批次（一个损坏的 PDF 不能停掉整条流水线）
- commit 失败：本批事件退回队列，下个窗口重试
- 索引损坏（打开报错）：界面给"重建索引"引导，不尝试自动修复

## 四、性能预算

| 指标 | 预算 |
|------|------|
| 单文件修改到可搜索 | < 3s（含 500ms 防抖） |
| 千文件批量变更（git checkout 场景） | < 30s 消化完 |
| 监听器常驻内存增量 | < 20MB |
| 启动对账 10 万文件 | < 30s，且不阻塞搜索 |

## 五、验收清单

1. 改一个 md 保存，3 秒内新内容可搜；删除该文件后立即搜不到。
2. 重命名文件，旧名不可搜、新名可搜、内容照常命中。
3. 删除整个子目录，其下全部文档从结果消失。
4. `git checkout` 切分支引发几百文件变更，30 秒内索引追平，期间搜索不卡。
5. 关掉 dowse，改/删/增几个文件，重新启动后不用手动重建，对账自动补齐。
6. 单元测试覆盖：防抖合并逻辑、重命名双边/单边事件、前缀删除。

## 六、明确不做（本里程碑）

- USN Journal（里程碑 6）；OCR 文件的增量（里程碑 4 一起考虑）
- 索引多个根的 UI 管理（托盘菜单暂时只支持加，删根走重建）
- 网络盘/可移动盘的监听（notify 在这些盘上不可靠，直接不注册监听，只靠启动对账）
